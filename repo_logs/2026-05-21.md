### 第二轮：Orchestrator 四层模型重构

**核心决策**：

- **D16**：Orchestrator 是唯一入口，星形拓扑核心。Web API 与 CLI 全部通过 `dispatch()` 进入，内部细节零对外暴露。**不采用状态机**——dispatch + phase 序列足够。
- **D10**：文件操作采用四层模型——Entry → Resolver → Planner → Primitive。每层职责单一、不越界。
- **D11-D13**：Entry 只拼 `TaskRequest`，`resolver_args` opaque；Resolver 只取资源不读状态；Planner 按 intent 自主决定 preflight。
- **D14-D15**：backup / apply / restore 各自独立 `*_ops.py`；严格 file-to-file，删除全部目录处理代码。
- **D17-D21**：compute 管线单独拆文件不改逻辑；preflight 独立模块；bakignore 保留 backup 专属；实施顺序「先建新文件，再改调用方，最后删旧代码」。

| 模块 | 文件 | 职责 |
|------|------|------|
| 核心 | `orchestrator/__init__.py` | `dispatch()` + `PipelineResult` |
| Entry | `orchestrator/entry.py` | `TaskRequest` + `Intent` |
| Resolver | `orchestrator/resolver.py` | `CleanContext` + 三种 Resolver |
| Planner | `orchestrator/planner_fileops.py` | `FileOpsPlan` + `plan_fileops()` |
| Preflight | `orchestrator/preflight.py` | apply / restore 门禁 |
| Compute | `orchestrator/compute_pipeline.py` | 映射生产（不动） |
| 共享 | `orchestrator/_common.py` | `PipelineResult` dataclass |
| Apply 原语 | `apply_ops.py` | `apply_entries()` |
| Restore 原语 | `restore_ops.py` | `restore_entries()` |
| Backup 原语 | `backup_ops.py` | `run_differential_backup()` |
