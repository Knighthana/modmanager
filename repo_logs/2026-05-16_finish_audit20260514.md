## 1. `useWorkspaceStore` —— 前端持久化唯一写者

**审计要求**：Pinia store 作为 storage 唯一写者，组件通过 store action 修改，store 负责 flush。

**落地**：`useAppStore`
- 文件：`frontend/src/stores/app.ts`（~80 行）
- 封装 persistence.ts 全部导出函数为 reactive action
- 10 个组件/页面 import 路径从 `../utils/persistence` → `../stores/app`
- 门禁：除 `stores/app.ts` 外无文件直接 import persistence.ts

**命名变更**：`useWorkspaceStore` → `useAppStore`。旧 audit 时 `workspace` 指 localStorage 中的 `modmanager:workspace` key；现在 workspace 已改义为后端工作区目录，app store 管的是 app 级 UI 状态。

---

## 2. `uiState` 并入 —— activeTab / sidebarCollapsed

**审计要求**：UI 状态聚合到统一位置，避免分散 key。

**落地**：通过 `useAppStore` 实现
- `sidebarCollapsed`：`appStore.sidebarCollapsed`（sessionStorage + localStorage 写穿）
- `activeTab`：`appStore.activeTab`（sessionStorage only）
- 与旧 audit 的区别：旧模型嵌套在 `uiState` 对象内，新模型平铺为独立 key——功能等价

---

## 3. 前端 Transport Abstraction

**审计要求**：`src/api/transport.ts` 统一入口，Tauri 迁移时仅改此层。

**落地**：transport.ts 统一出入口
- `transport.ts` 新增 re-export：`apiPost`、`invoke`（别名）、`streamSse` + 全部类型
- 12 个文件 import 路径从 `api/client` 或 `api/sse` → `api/transport`
- Tauri 迁移路径：仅替换 transport.ts 中两个 re-export 的源即可
- 门禁：除 transport.ts 自身外，无文件直接 import api/client 或 api/sse
